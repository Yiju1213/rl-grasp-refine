# v3：当前系统事实与边界

本文件用于对齐 **当前仓库实际实现**，覆盖 `v2.md` 中已经过时的 env / scene 描述。

它的角色是：

- 记录当前系统已经落地的事实
- 说明当前系统的真实边界
- 明确哪些能力仍是调试入口或实验入口，而不是正式 runtime 方案
- 记录当前已知但暂未修正的契约问题

本文件优先级高于 `v2.md` 中与 env / scene 现状冲突的描述。

---

## 1. 当前系统的核心形态

当前系统已经固定为：

- **dataset-backed**
- **single-step**
- **grasp refinement RL**

每个 episode 的语义是：

1. 从 tactile-extended 数据集采一个 `before`
2. 用 `before` 对应的 grasp / hand / object metadata 重建 reset 场景
3. policy 输出一个 6-DoF delta action
4. env 对 grasp 做一次 refinement
5. 重建 refined grasp
6. 采集 `after`
7. 执行一次 release trial
8. `done=True`

因此，当前 episode 仍然是：

`reset -> 一次 action -> trial -> done`

并且：

- 当前 `after` **不会**作为下一轮的 `before`
- 下一轮总是重新从数据集采样

---

## 2. 当前 env / scene 的真实状态

当前 `PyBulletScene` 已经不是旧的 heuristic scene。

它已经具备以下真实行为：

- 使用真实 hand URDF
- 使用真实 object URDF
- 使用 TACTO 传感器
- `before` 直接读取数据集图像与 metadata
- `after` 来自当前仿真的实时采集
- `reset` 和 `refine` 共享一套抓持重建流程
- `release` 通过真实仿真步进和接触状态判定成功/失败

当前 `PyBulletScene` 的 reset / refine 语义是：

1. `hand.reset()`
2. 物体重置到指定 object pose
3. 建立 object constraint
4. 手 base 放到目标 hand pose
5. 位置闭合直到接触成立或 timeout
6. 力闭合直到达到目标力或失败
7. refine 成功后才进入 release

因此，当前系统已经不是：

- target pose 误差 heuristic 判成败
- box 近似 object
- fake after observation

---

## 3. 当前 scene lifecycle 决策

当前 `PyBulletScene` 采用的是：

- `hand` 常驻
- `tacto_sensor` 常驻
- **不做** episode 级 `resetSimulation()`
- object 按需切换

具体 object 切换策略：

- 若新 sample 与当前 `object_id` 相同：
  - 复用当前 object body
- 若新 sample 与当前 `object_id` 不同：
  - 从 tacto 中移除旧 object
  - `removeBody(old)`
  - `spawn_object(new)`
  - 将新 object 重新挂到 tacto

当前 **没有**把 preload-all-objects 放进正式 scene runtime。

`--preload-all-objects` 只存在于：

- `scripts/sanity_check_env.py`

它是调试 / 对比实验入口，不是正式 env 设计的一部分。

## 4. 当前资产边界

当前 runtime 直接依赖的 scene 资产只有：

- hand Python 逻辑
- hand URDF
- tacto background
- tacto config
- object URDF root

当前 **table 已经从 runtime scene 主逻辑移除**。

说明：

- 代码仓库里可以仍然保留 table 资产文件
- 但当前 `PyBulletScene` 不再 spawn table
- table 也不是当前 reset / refine / release 语义的一部分

## 5. 当前数据集采样事实

当前 `DatasetSampleProvider` 已采用：

- **全局 grasp 级扁平打乱**

即：

- 初始化时构建全量 `(object_id, global_id)` pair
- 每个 epoch 对全量 pair 做 shuffle
- `sample()` 每次弹出一个 pair

这意味着：

- 单 env 下，episode 级别也会跨 object 切换
- 当前采样不是“长时间停留在同一 object”

另外，`source.legacy_drop_success` 当前仍被保留在 sample metadata 中，但其角色已经明确限定为：

- 仅可能用于离线校验 / 对账 / 兼容分析
- **不会参与训练逻辑**
- **不会参与 reward / trial / PPO / calibrator**

## 6. 当前观测事实

当前 env 输出的 `RawSensorObservation` 仍固定分为：

- `visual_data`
- `tactile_data`
- `grasp_metadata`

当前实际提供的核心字段包括：

- `visual_data.rgb / depth / seg / view_matrix / proj_matrix`
- `tactile_data.rgb / depth / proj_matrix / sensor_poses_world / camera_distance_to_gel_m / contact_map / contact_force`
- `grasp_metadata.grasp_pose / object_pose_world / source_object_id / source_global_id / observation_stage / segmentation_ids / gel_pose_world / observation_valid`

当前 env **不再直接输出**：

- `visual_data.point_cloud`
- `visual_data.distance_to_edge`

当前责任边界是：

- `point_cloud` 由 adapter 占位生成
- `coverage_ratio / edge_proximity` 由 contact semantics 占位生成

---

## 7. 当前 visual camera 的真实含义

当前数据集保存的 visual camera：

- `view_matrix`
- `visual_proj_matrix`

其物理几何含义是：

- `view_matrix` 是 **世界坐标系到相机坐标系** 的外参矩阵
- 它虽然最初是通过 hand pose 与观察方向构造出来的
- 但保存结果本身已经是 **世界系相机矩阵**

当前 repo 在 live after observation 中的做法是：

1. 从 sample 的 `view_matrix` 反推出 sample camera pose
2. 与 sample 的 `pre_grasp.hand_pose_world` 组合
3. 求出 hand-to-camera 相对位姿
4. 再用当前 hand pose 重建当前 visual 相机

因此，当前系统对 visual camera 的理解是：

- 保存的是世界系外参
- 但它与 hand pose 强相关

## 8. 当前 Bullet client 语义

当前代码明确采用以下假设：

- **每个进程只维护一个 Bullet client**
- 当前 client id 约定为 `0`
- RL training 的多 env 预期形态是 **多进程**，不是同一进程里维护多个 Bullet world

这意味着当前代码**不以单进程多 client 为目标**。

在这个前提下：

- `client_id=0` 的硬编码是刻意的简化
- 它和当前规划中的多进程训练方式兼容
- 但它**不支持**“同一进程里并行维护多个 Bullet client / 多个 scene world”这一类设计

当前 [sanity_check_env.py](./../scripts/sanity_check_env.py) 已经不仅是最小单步脚本，还承担了 env / scene 调试入口角色。

它当前支持：

- GUI 模式

其中：

- `--preload-all-objects` 仅用于实验 “预加载全部 object 是否影响 GUI / reset 行为”
- 它**不代表正式 scene 方案**

## 10. 当前已知问题与未对齐项

### 10.1 `env_side_adapter_contract.md` 仍有一个潜在契约问题

当前 [env_side_adapter_contract.md](./env_side_adapter_contract.md) 对：

- `tactile_data.sensor_poses_world`
- `grasp_metadata.gel_pose_world`

的描述仍写成“总是具有完整 `left/right.position/quaternion` 结构”。

但当前真实实现里，在 invalid-after 路径下：

- `sensor_poses_world = {"left": None, "right": None}`
- `gel_pose_world = {"left": None, "right": None}`

也就是说：

- 当 `observation_valid=False` 时
- 这两个字段目前允许为 `None`

这个问题当前**先记录，不在本轮直接改 contract 文档**。

### 10.2 GUI 模式下的 object 渲染问题仍未根治

当前已知现象是：

- DIRECT 模式下，object 可被稳定拍摄
- GUI 模式下，PyBullet ExampleBrowser 主视口/预览对部分 object 的 visual mesh 渲染不稳定
- 但 AABB、TACTO、以及 DIRECT 采样都表明 physics body 本身仍在

因此当前判断是：

- 训练主路径仍然以 DIRECT 为准
- GUI 问题更像 Bullet GUI / OpenGL visual path 的问题，而不是 scene physics 正确性问题

## 11. 当前建议的理解方式

如果只用一句话描述当前系统，应理解为：

**一个以 tactile-extended 数据集为 before、以 PyBullet+TACTO 实时仿真为 after 的 single-step grasp refinement RL 最小系统。**

它已经具备：

- 真实 object / hand / tacto 资产
- 真实 reset / refine / release 编排
- 可运行训练链路

它仍然保留的最小化/占位部分主要在：

- adapter 的 point cloud 占位
- contact semantics 的占位公式
- evaluation / ablation 的薄实现
- GUI 路径的不稳定调试问题

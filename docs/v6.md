## 2. 当前正式系统定义

### 2.1 任务与 episode 语义

当前正式系统是一个 **dataset-backed single-step in-contact grasp pose refinement** 系统。

其 episode 语义固定为：

- 从数据集采样一个 `before` 抓取状态；
- 进行一次 6-DoF corrective action；
- 在仿真中重建 refined grasp；
- 采集 `after` 观测；
- 执行一次 release trial；
- episode 结束，不把当前 `after` 继续滚入下一轮。

因此，当前正式系统不是长时序 manipulation policy，也不是多步 regrasp 规划器，而是围绕**单步抓取位姿修正**组织的感知-校准-奖励-RL 闭环。

### 2.2 状态、动作与默认 policy observation

当前动作定义仍为单步 6 维归一化动作，经 `ActionExecutor` 解码为：

- 3 维平移增量；
- 3 维旋转增量。

当前正式状态默认按论文主表达使用：

- `s = [z || t]`

其中：

- `z` 是由显式对齐的 visuo-tactile perception 主链提取的 latent feature；
- `t` 是 `contact_semantic = [t_cover, t_edge]`。

当前默认配置中：

- `configs/model/actor_critic.yaml` 的 `policy_observation.preset: paper`
- 对应 observation 组件仅包含 `latent_feature + contact_semantic`

这意味着当前默认正式观测已经不是旧的 `current` 组合，而是论文主表达下的 `paper` preset。

### 2.3 感知、接触语义与稳定性链

当前正式感知主链由以下部分组成：

- dataset / scene 提供原始 visual 与 tactile 观测；
- `ObservationBuilder` 统一构造结构化 observation；
- `FeatureExtractor.encode(raw_obs)` 生成 latent feature；
- `ContactSemanticsExtractor` 生成 `t_cover` 与 `t_edge`；
- runtime 若已返回 raw stability logit，则直接使用；
- 否则由 `StabilityPredictor` 基于 latent feature 预测 raw logit。

当前 `contact_semantic` 已是正式状态量，不是调试字段：

- `t_cover`：active contact 占有效 tactile 点的比例；
- `t_edge`：active tactile 点到 sensor 边界的归一化距离均值。

当前稳定性链仍是：

- raw stability logit；
- 两参数 online logistic calibration；
- calibrated probability；
- posterior covariance trace。

也就是说，当前正式系统中稳定性信号链仍为：

- `raw logit -> calibrated probability -> posterior uncertainty`

### 2.4 reward 定义

当前正式 reward 仍由三项组成：

- `R_drop`
- `R_stability`
- `R_contact`

其工程语义为：

- `R_drop`：由 release trial 成功/失败给出基础奖励；
- `R_stability`：使用 refined 前后 calibrated probability 的差值，并由 posterior trace 衰减；
- `R_contact`：对 refined 后的 `t_cover / t_edge` 做 soft-hinge penalty。

默认 reward 配置位于 `configs/env/grasp_refine_env.yaml`，默认三项权重均开启。

### 2.5 默认训练范式

当前正式训练默认不是单环境串行，而是：

- PPO actor-critic；
- 多进程异步 rollout；
- 单步 transition buffer；
- rollout 后统一 PPO update；
- rollout 后统一 calibrator update。

默认配置中：

- `configs/rl/ppo.yaml` 的 `num_envs: 12`

因此，当前正式训练默认走 `SubprocAsyncRolloutCollector`，而不是单 env 同步采样。

## 3. 当前默认训练与验证主链

### 3.1 入口与配置解析

当前默认训练入口是：

- `scripts/train.py`

运行时流程是：

1. `train.py` 读取 experiment 配置。
2. `scripts/_common.py` 加载 config bundle。
3. `src/runtime/experiment_config.py` 将 experiment 级设置下沉到 env、perception、calibration、rl。
4. `build_actor_critic()` 与 `build_env()` 按解析后的 bundle 构造正式组件。

当前默认 experiment 配置是：

- `configs/experiment/exp_debug.yaml`

其中默认已开启：

- validation
- object split
- best checkpoint
- metric profile = `paper`

### 3.2 object split 与 train/validation 数据边界

当前正式训练主链支持对象级切分。

`src/runtime/object_splits.py` 会根据 experiment 配置解析：

- `train_object_id_range`
- `holdout_object_id_range`
- `val_object_count` 或 `val_object_ratio`
- `split_seed`

从而得到：

- `train_ids`
- `val_ids`
- `test_ids`

默认训练行为是：

- rollout 训练样本只从 `train_ids` 采样；
- validation rollout 只从 `val_ids` 采样；
- `test_ids` 被保留为 holdout 测试池，不参与默认训练更新。

因此，当前正式系统已经具备对象级 train/val/test 划分，而不是把所有对象混在同一个 rollout 池里。

### 3.3 默认训练闭环

训练 iteration 的默认顺序如下：

1. 主进程准备 actor snapshot 与 calibrator state。
2. collector 将 snapshot 广播到 worker。
3. worker 从训练对象池中异步收集 valid episodes。
4. 主进程聚合 batch，计算 return 与 advantage。
5. PPO 主进程执行参数更新。
6. 主进程使用本轮 `raw_logit_after + drop_success` 更新 calibrator。
7. 若开启 validation，则使用 post-update calibrator state 进行 validation rollout。
8. 记录日志、保存样本指标、维护 best checkpoint。
9. iteration 结束后清空 buffer，进入下一轮。

当前默认训练闭环中的正式组件包括：

- `Trainer`
- `PPOAgent`
- `RolloutBuffer`
- `SubprocAsyncRolloutCollector`
- `OnlineLogitCalibrator`

### 3.4 validation 主链

当前 validation 不是外部脚本级附加能力，而是训练主链内的正式组成部分。

默认配置中：

- `configs/experiment/exp_debug.yaml` 的 `validation.enabled: true`
- 默认 best metric 为 `validation/outcome/success_lift_vs_dataset`

validation 的默认语义是：

- 使用 validation object split；
- 不参与 PPO update；
- 不参与 calibrator update；
- 复用当前 actor 参数；
- 使用训练 batch 更新后的 calibrator state；
- 输出与训练侧同命名空间对应的 `validation/*` 聚合指标。

当前 validation 既支持：

- `validation_env`
- 也支持独立的 `validation_collector`

因此，validation 已经是默认正式系统中的一条独立闭环，而不是仅靠训练 rollout 指标做模型选择。

### 3.5 checkpoint、恢复与配置快照

当前正式训练主链会维护以下产物：

- `final.pt`
- 可选 `best.pt`

默认 best checkpoint 规则来自：

- `configs/experiment/exp_debug.yaml`

默认设置为：

- `metric: validation/outcome/success_lift_vs_dataset`
- `mode: max`
- `filename: best.pt`

此外，训练开始时会自动保存本次实验所用的配置快照到输出目录下的 `configs/`。

当前训练也正式支持从 checkpoint 恢复：

- `scripts/train.py --resume-from <checkpoint>`

这意味着当前正式主链已经把：

- checkpoint 持久化
- best model 选择
- resume 继续训练
- config snapshot

都纳入了正式运行闭环。

### 3.6 collector 运维能力

当前 collector 不仅负责异步采样，也承担正式运维能力。

当前主链已支持：

- `scene_rebuild_every_n_iterations`
- `worker_recycle_every_n_iterations`
- `worker_recycle_slots_per_event`
- `worker_recycle_enable_standby_prefetch`
- `worker_recycle_prefetch_count`

其语义是：

- 可按 iteration 触发 worker scene rebuild；
- 可按 iteration 主动回收最老 worker；
- 可在当前采样期间后台预热 standby worker；
- 下轮 recycle 时优先切换到 ready 的新 worker。

这些能力当前已经在训练主链和 validation 主链中被正式消费，不应再视为附属调试功能。

## 4. 当前可配置接口与运行产物

### 4.1 experiment 层正式接口

当前应视为正式 experiment 级接口的字段包括：

- `train_object_id_range`
- `holdout_object_id_range`
- `split_seed`
- `val_object_count`
- `val_object_ratio`
- `validation.*`
- `ablation.id`
- `scene_rebuild_every_n_iterations`
- `worker_recycle_every_n_iterations`
- `worker_recycle_slots_per_event`
- `worker_recycle_enable_standby_prefetch`
- `worker_recycle_prefetch_count`

其中当前默认正式实验以 `configs/experiment/exp_debug.yaml` 为参考实现。

### 4.2 actor-critic 层正式接口

当前应视为正式公开接口的 actor-critic 相关字段包括：

- `architecture.type`
- `policy_observation.preset`
- `policy_observation.components`

当前默认正式基线为：

- `architecture.type: plain`
- `policy_observation.preset: paper`

当前系统也支持可选的：

- `architecture.type: latent_first_late_fusion`

但这不是默认正式基线。

### 4.3 ablation 接口

当前 `src/runtime/experiment_config.py` 正式支持的 `ablation.id` 包括：

- `baseline`
- `wo-tac-rwd`
- `wo-stb-rwd`
- `wo-onl-cal`
- `wo-tac-sem-n-rwd`

这些 preset 会在运行前展开为：

- reward 权重覆盖；
- calibration 开关覆盖；
- policy observation 组件覆盖。

因此，ablation 当前不是外部实验脚本约定，而是正式配置接口的一部分。

### 4.4 reproducibility 与产物接口

当前正式系统中与可复现性直接相关的能力包括：

- `snapshot_experiment_configs()`
- `scripts/export_single_file_config.py`
- `src/utils/single_file_config.py`

其作用分别是：

- 保存本次训练真实使用过的 yaml 快照；
- 将 experiment bundle 导出为单文件配置；
- 在嵌套配置布局下恢复有效配置结果。

当前默认训练输出目录下的主要产物包括：

- `metrics.jsonl`
- `run.log`
- TensorBoard event files
- `checkpoints/final.pt`
- `checkpoints/best.pt`
- 可选 `episode_metrics.jsonl`
- `configs/` 配置快照

### 4.5 monitoring 接口

当前默认监控接口遵循以下规则：

- 默认 `logging.metric_profile: paper`
- 若开启 validation，同类聚合信号使用 `validation/` 前缀
- `ppo/*` 只存在于训练侧，不进入 `validation/*`

因此，当前默认正式监控视图是：

- 一组 paper profile 下的核心训练信号；
- 再加一组同口径的 `validation/*` 信号。

更完整的指标解释可参考 `docs/signal.md`。

## 5. 当前已知边界与非目标

### 5.1 calibrator / reward 仍是 snapshot 语义

当前 reward 计算使用的是 rollout 时广播到 env / worker 的 calibrator snapshot。

这意味着：

- `R_stability` 使用的是采样时的 posterior snapshot；
- calibrator 的参数更新发生在 batch 收集结束后；
- 当前实现是因果一致的 online 版本，但不是 same-batch posterior reward 语义。

这是当前正式系统的已知边界，不是实现缺失。

### 5.2 `t_edge` 是工程化定义

当前 `t_edge` 已有正式实现，但其定义是：

- 基于 tactile sensor 本地 2D 网格；
- 统计 active contact 到最近边界的归一化距离均值。

这个定义与当前系统目标一致，也已进入正式状态与 reward 链，但它仍是工程化边界距离定义，而不是唯一可能的物理解释版本。

### 5.3 `scripts/evaluate.py` 不是默认 model-selection 主链

当前仓库存在轻量评估入口：

- `scripts/evaluate.py`

它可用于读取 checkpoint 并跑若干 episodes，但当前默认正式模型选择不依赖这条脚本链，而是依赖训练主链内的 validation 指标与 best checkpoint 规则。

因此，`evaluate.py` 当前应视为轻量评估工具，而不是正式训练主链中的 model-selection authority。

### 5.4 可选架构存在，但默认正式基线仍是 plain

当前系统支持：

- `plain`
- `latent_first_late_fusion`

两种 actor-critic 架构。

但当前默认正式配置仍是：

- `architecture.type: plain`

因此，late fusion 是可选能力，不应被表述为当前默认正式基线。

### 5.5 本文档不承诺训练结果结论

本文档描述的是当前正式系统规范，不对实验效果、泛化结论或阶段性分析做承诺。

若需要查看训练结果判断，应参考：

- `docs/train-result/`

若需要查看更完整的参数与信号定义，应参考：

- `docs/config.md`
- `docs/signal.md`

## 6. 当前默认系统的简要结论

综合当前代码与默认配置，当前正式系统可以概括为：

- 一个 dataset-backed 的 single-step grasp pose refinement 系统；
- 默认状态已按 `paper` preset 使用 `latent_feature + contact_semantic`；
- 默认 reward 为 `drop + stability + contact` 三项；
- 默认训练采用 12 worker 的异步 PPO rollout；
- 默认训练内置对象级 train/validation 切分与 validation 闭环；
- 默认 best checkpoint 基于 validation 的 relative lift；
- 默认日志、配置快照、checkpoint、resume 与 collector 运维能力都已进入正式主链。

因此，当前仓库中的正式训练系统，已经不只是“能跑 PPO 的单步环境”，而是一套包含默认训练、默认 validation、模型选择、配置固化与多进程运维能力的完整正式训练链。

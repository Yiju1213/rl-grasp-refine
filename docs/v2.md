# v2：当前系统现状盘点与开发路线图

本文件基于当前仓库代码、配置、测试文件、[v1.md](./v1.md) 与 [implementation_assumptions.md](./implementation_assumptions.md) 整理。
它的角色是“现状盘点 + 路线图”，不是新的实现规范，也不替代 `v1.md`。

---

## 1. 系统目标

当前系统的目标是：

构建一个**可运行的 single-step grasp refinement RL 最小框架**，让以下主链路已经可以闭环：

1. 从结构化 observation 构造策略输入。
2. 由 policy 输出一个 6-DoF 小幅 refinement action。
3. 环境执行 refinement，得到 after observation、trial outcome 与 reward。
4. calibrator 对 stability logit 做在线校准。
5. PPO trainer 基于 rollout 做策略更新，并保存日志与 checkpoint。

从 `v1.md` 的原始意图看，这套系统对应的是论文系统的**第一版工程骨架**，目的是先把模块边界、数据契约、最小训练路径打通，而不是立即落地完整论文方法。

因此，当前系统**不是**论文完整系统。它目前不包含：

- 真实传感器接入
- 真实 grasp 物理任务与真实机器人/夹爪资产
- 论文级 backbone / stability predictor
- 完整的 evaluation / ablation 实验体系
- 具有科研意义的训练结果

## 2. 当前系统完成情况

### 2.1 总体判断

当前仓库已经不是空骨架，而是一个**能跑通最小训练闭环**的工程框架：

- 有完整目录结构、配置、脚本入口与模块拆分
- 有可运行的 env / perception / calibration / RL 主链路
- 有自动化单元测试与最小集成测试
- 有一次 smoke 级训练运行产物：`outputs/exp_debug/run.log`、`metrics.jsonl`、`checkpoints/final.pt`

但同时，这个闭环中有多处实现仍是**最小占位版本**，其主要价值是验证工程路径，而不是替代论文真实方法。

| `evaluation/` | 部分实现 | `metrics` 与 `Evaluator` 可用 | `AblationRunner` 仍为 stub；评估体系仍很薄 |

### 2.3 各模块目前分别完成了什么

#### `structures / utils / config / scripts`

这一层已经完整，作用是把整个工程变成一个可运行项目，而不是散模块。

- `structures/` 提供了环境、感知、训练之间共享的结构化数据语言。
- `utils/tensor_utils.py` 明确了 observation 进入 RL 网络前的唯一 flatten 边界。
- `utils/logger.py` 与 `utils/checkpoint.py` 支持训练过程记录与结果保存。
- `configs/` 提供了最小 debug 级别配置组合。
- `scripts/` 提供训练、评估、sanity check、单条 rollout 调试、calibrator 调试入口。

#### `envs`

这一层已经实现了 single-step env 主体。

- `GraspRefineEnv.reset()` 会 sample 初始状态、设置初始 grasp、采 raw obs、构建 before observation。
- `GraspRefineEnv.step()` 会 decode action、得到 refined pose、执行 trial、构建 after observation、调用 calibrator、计算 reward、返回 `done=True`。
- `ActionExecutor` 已固定当前动作空间：`[dx, dy, dz, rx, ry, rz]`，旋转使用 rotation vector。
- `RewardManager` 已实现 proxy reward。
- `SingleStepTermination` 明确 single-step episode。

当前限制在于：

- `PyBulletScene` 虽然接了 PyBullet，但 grasp 成败仍由 target pose 误差 heuristic 决定。
- 当前 raw observation 也是合成的 point cloud / tactile 信号，不是真实传感器数据流。

#### `perception / models`

这一层已经把“raw observation -> latent feature / contact semantic / raw stability logit”打通。

- adapter 已把 raw scene observation 适配到 backbone/predictor 输入。
- `FeatureExtractor` 与 `StabilityPredictor` 已封装冻结推理路径。
- `ContactSemanticsExtractor` 已输出二维接触语义：coverage 与 edge proximity。
- `SGAGSNEncoder` 和 `StabilityHead` 都可运行，但它们是工程占位实现。

这意味着：

- 当前系统已经有 perception 层边界与数据流。
- 但它还没有论文真实模型，也没有真实预训练权重、真实特征构造或真实多模态对齐逻辑。

#### `calibration`

这一层已具备最小 online calibration 能力。

- `predict()` 输入 raw logit，输出 calibrated probability 与 uncertainty。
- `update()` 在 rollout 级别使用 `raw_logit_after -> drop_success` 更新参数。

当前限制在于：

- 这是两参数 logistic calibration 的近似版本。
- uncertainty 来自近似 posterior covariance，不是论文级严格不确定度建模。

#### `rl`

这一层已经构成最小 PPO 训练链路。

- `PolicyNetwork` 与 `ValueNetwork` 都是可训练 MLP。
- `ActorCritic` 已统一 act / evaluate_actions 接口。
- `RolloutBuffer` 能同时保存 PPO 需要的标量 reward 与 calibrator 需要的 raw logits。
- `PPOAgent` 已实现 clipped objective、value loss、entropy bonus。
- `Trainer` 已实现 collect rollout、PPO update、calibrator update、logging。
- `DummyVecEnvWrapper` 已预留同步多 env 接口。

当前限制在于：

- `num_envs` 默认仍是 1。
- `DummyVecEnvWrapper` 只是同步封装，不是实际并行采样基础设施。

#### `evaluation`

这一层只完成了最小评价出口。

- `metrics.py` 已提供 success rate / average reward / average stability gain。
- `Evaluator` 已能跑 learned policy 与 random policy。

当前限制在于：

- 还没有 baseline 管理、实验矩阵、统计聚合、多 seed、置信区间输出。
- `AblationRunner` 仍然是 stub。

### 2.4 当前系统如何协同

当前系统的主训练调用链已经明确：

`train.py`
-> `load_experiment_bundle()`
-> `build_env() / build_actor_critic()`
-> `Trainer.train()`
-> `Trainer.collect_rollout()`
-> `env.reset()`
-> `ObservationBuilder.build()`
-> `actor_critic.act()`
-> `env.step()`
-> `scene.apply_refinement() / scene.run_grasp_trial()`
-> `observation_builder.build(after)`
-> `calibrator.predict(before/after)`
-> `reward_manager.compute()`
-> `buffer.add(...)`
-> `PPOAgent.update()`
-> `Trainer.update_calibrator()`
-> `logger.log_dict(...)`
-> `save_checkpoint(...)`

这条链路说明：

- 当前工程分层已经闭环。
- 后续替换真实 scene、真实 perception、真实 reward/calibrator 时，只要尽量不打破这些边界，整体训练脚本层可以保留。

## 3. 当前测试工作已经完成了什么，以及这意味着什么

### 3.1 当前自动化测试覆盖

当前 `tests/` 已覆盖以下内容：

- `tests/test_env.py`
  - 验证 `env.reset()` 返回 `Observation`
  - 验证 `env.step()` 返回合法四元组
  - 验证 single-step episode 下 `done=True`
- `tests/test_reward.py`
  - 验证 reward breakdown 存在
  - 验证 `total == weighted components`
- `tests/test_calibrator.py`
  - 验证 calibrator 可 `predict`
  - 验证 `update()` 后参数会变化
- `tests/test_policy_io.py`
  - 验证 policy/value 网络输出 shape
  - 验证 `ActorCritic.act()` 接口 shape
- `tests/test_single_step_pipeline.py`
  - 验证 env -> policy -> step 单条链路可运行
  - 验证 trainer 可完成单轮训练
  - 验证 `DummyVecEnvWrapper` 的输入输出维度与顺序

这些自动化测试以 `FakeScene` 为基础，因此它们测试的是：

- 工程接口正确性
- 主链路是否能跑通
- 当前最小 PPO 管线是否具备基本可执行性

它们**不是**在验证真实物理行为。

### 3.2 当前 smoke 验证

当前仓库还提供并已经用于最小 smoke 检查的脚本：

- `scripts/sanity_check_env.py`
- `scripts/debug_calibrator.py`
- `scripts/debug_single_rollout.py`
- `scripts/train.py`

其中，当前仓库已有一次 5-iteration smoke training 的结果产物：

- `outputs/exp_debug/run.log`
- `outputs/exp_debug/metrics.jsonl`
- `outputs/exp_debug/checkpoints/final.pt`

这些产物说明：

- 训练脚本至少完成过一次从 collect rollout 到 update、再到 log/checkpoint 的全流程运行。
- 这只能说明“最小闭环已跑通”，不能说明训练已经有效。

### 3.3 这意味着哪些测试之后不需要再单独手工做

以下内容**不需要再作为额外手工 sanity check 反复做**：

- `reset()/step()` 是否返回基本合法结构
- single-step env 是否 `done=True`
- reward breakdown 是否能组成 total
- calibrator 的 `update()` 后参数是否变化
- policy/value/actor_critic 的基础输入输出 shape 是否正确
- trainer 是否能至少跑完单轮最小训练
- `DummyVecEnvWrapper` 的最小 list-based 接口是否成立

但这不意味着这些测试可以删除。

更准确的含义是：

- 这些内容已经适合作为**自动 regression tests** 固定下来。
- 后续不应再依赖人工重复点击或 print 检查去验证它们，除非你修改了这些模块的接口契约。

### 3.4 当前测试没有证明什么

当前测试与 smoke 运行**没有证明**以下事项：

- 没有证明真实物理抓取有效
- 没有证明真实 perception/backbone 有效
- 没有证明 raw observation 与真实传感器契约正确
- 没有证明 reward/calibrator 已与论文严格一致
- 没有证明训练结果具有科研意义
- 没有证明多 env、并行采样、evaluation/ablation 已经可用

## 4. 当前已冻结的接口事实

以下内容虽然最初来自 `v1.md` 中未完全明确的部分，但在当前代码里已经被实际实现并固定下来，应视为当前系统事实：

### 4.1 数据契约

- `Observation.grasp_pose` 使用 `GraspPose`，不是裸 `np.ndarray`
- `Transition.reward` 保存的是 scalar reward
- 分项 reward 不进入 PPO 主 reward 字段，而是保存在 `StepInfo.extra["reward_breakdown"]`
- `StepInfo.extra` 当前固定携带：
  - `reward_breakdown`
  - `raw_logit_before`
  - `raw_logit_after`
  - `trial_metadata`

### 4.2 动作与姿态契约

- action 顺序为 `[dx, dy, dz, rx, ry, rz]`
- translation 和 rotation 都先在 `[-1, 1]` 内归一化，再映射到物理边界
- 当前 rotation 表示固定为 rotation vector
- 当前动作边界默认来自 config：
  - translation 每轴 `±0.01m`
  - rotation 每轴 `±0.1rad`

### 4.3 observation -> tensor 契约

`observation_to_tensor()` 当前唯一合法 flatten 顺序为：

`latent_feature | contact_semantic | grasp_pose.position | grasp_pose.rotation | raw_stability_logit`

这意味着：

- 前面的 env / perception / reward 模块依然是结构化对象通信
- 只有进入 RL 网络时才拍扁

### 4.4 calibrator 与 rollout 契约

- calibrator 当前只使用 `raw_logit_after -> drop_success` 做在线更新
- buffer 必须同时保存 PPO 训练所需标量和 calibrator 所需 raw logits
- uncertainty 当前是近似量，不应被误写成最终科学定义

### 4.5 vec env 契约

- `DummyVecEnvWrapper.reset()` 返回 `list[Observation]`
- `DummyVecEnvWrapper.step()` 返回：
  - `list[Observation]`
  - `np.ndarray rewards`
  - `np.ndarray dones`
  - `list[StepInfo]`
- 当前实现是同步 dummy wrapper，不是实际并行后端

## 5. 下一阶段开发板块与推荐顺序

下一阶段建议分为五个大板块，推荐顺序为：

`A -> B -> C -> D -> E`

原因很简单：

在真实任务/场景契约没有冻结之前，后面的 perception、reward、evaluation 都很容易返工；而在真实 env/perception/reward 没对齐之前，跑出来的训练结果也很难有实际意义。

### A. 真实任务 / 场景 / 样本契约

目标：

- 用真实对象、真实 grasp 场景、真实初始 grasp、真实 raw observation / trial 契约替换当前 heuristic `PyBulletScene`
- 冻结 `sample_cfg` 的真实 schema

优先需要完成的内容：

- 明确 object / grasp / scene 资产来源
- 明确 reset 时需要哪些输入字段
- 明确 `get_raw_observation()` 的真实输出结构
- 明确 `run_grasp_trial()` 的真实执行和标签来源

为什么先做：

- 这是当前系统与真实论文任务之间最大的断层
- 如果这里不先定，后面 perception 与 reward 都只能围绕假的输入输出开发

### B. 真实 perception / backbone / stability predictor

目标：

- 用论文真实模型、已有代码或已有权重，替换当前 summary-based placeholder

优先需要完成的内容：

- adapter 输入格式与真实模型对齐
- backbone forward 接口与 checkpoint 加载
- predictor 输入输出与真实 stability logit 定义对齐
- 冻结“完全冻结 / 部分微调 / 联合训练”的策略

为什么排第二：

- scene 与 raw observation 契约先定下来后，才能知道 perception 真正吃什么输入
- 否则会先写一版很快失效的 adapter 和 predictor glue code

### C. reward / calibrator 与论文对齐

目标：

- 把当前 proxy reward 与最小 calibrator 替换为论文定义或更接近论文定义的实现

优先需要完成的内容：

- 明确最终 `R_drop / R_stability / R_contact` 公式
- 明确 calibrator 的监督标签、更新频率、更新窗口、uncertainty 含义
- 明确 before/after 在 reward 和 calibration 中的精确角色

为什么排第三：

- reward 和 calibrator 强依赖真实 scene 与真实 predictor 输出
- 在 A、B 没定之前先精化这部分，通常会返工

### D. 第一轮真实 training + evaluation

目标：

- 在真实 env、真实 perception、真实 reward/calibrator 下，拿到第一轮“有意义”的训练与评估结果

优先需要完成的内容：

- 跑第一轮短训练
- 用 `evaluate.py` 跑 learned policy 与 random policy 对比
- 明确最小 baseline
- 输出更可靠的 summary metrics

为什么排第四：

- 只有 A/B/C 都对齐后，这一轮训练才值得分析

### E. 扩展与实验化

目标：

- 提升训练效率、实验覆盖和工程化程度

优先需要完成的内容：

- 真正启用 vec env / 并行采样
- 补 `AblationRunner`
- 增加多 seed 训练
- 完善日志、实验目录与结果汇总

为什么放最后：

- 这些工作是重要的，但不是最快得到第一轮真实训练结果的路径

## 6. 未来需要的测试工作

未来测试建议按“验证目的”来分，而不是只按文件分。

### 6.1 真实场景 integration tests

用于验证：**物理真实性**

建议测试内容：

- asset loading 是否稳定
- `reset_scene()` 是否生成正确场景
- `apply_refinement()` 是否正确作用于场景状态
- `run_grasp_trial()` 的输出标签是否合理
- 同一 seed 下 scene 是否足够稳定可复现
- 常见失败模式是否能被正确检测和报告

### 6.2 perception / model tests

用于验证：**模型真实性**

建议测试内容：

- adapter 输入输出 shape 与字段映射
- backbone / predictor checkpoint load
- 固定输入下输出是否回归稳定
- `freeze=True` 时是否真的 `eval()` + `no_grad()`
- 替换真实模型后，latent/logit 的分布是否基本合理

### 6.3 reward / calibrator numeric tests

用于验证：**公式正确性**

建议测试内容：

- reward 公式精确数值测试
- 单调性测试
- clipping / boundary 行为测试
- uncertainty 输出行为测试
- calibrator update 后的趋势测试
- small batch / extreme logits / all-same labels 下的稳定性测试

### 6.4 training tests

用于验证：**训练稳定性**

建议测试内容：

- 参数是否真的更新
- loss 是否为有限值
- 短训练是否可重复
- save / load checkpoint 后训练是否可恢复
- 单 env 与 vec env 路径是否都能稳定 collect rollout

### 6.5 evaluation tests

用于验证：**实验输出可信性**

建议测试内容：

- `metrics.py` 计算结果是否正确
- learned / random policy 路径是否都能跑通
- baseline 配置是否正确加载
- future ablation 配置矩阵是否正确展开

### 6.6 multi-env tests

用于验证：**并行采样正确性**

建议测试内容：

- episode 计数是否正确
- batch 对齐是否正确
- `obs/action/reward/info` 顺序是否一致
- raw logit 与 drop label 是否没有错位
- future real vec env 与当前 dummy wrapper 接口是否兼容

## 7. 当前阶段的结论

当前系统已经完成了一个**能训练、能调试、能自动回归测试的最小 RL grasp refinement 工程框架**。

它已经足够支持下一阶段开发，但它的价值主要体现在：

- 模块边界已经成立
- 数据契约已经基本冻结
- 最小训练路径已经跑通
- 基础工程测试已经具备

它还没有达到的，是：

- 真实任务闭环
- 真实论文模型闭环
- 真实 reward/calibration 闭环
- 有科研意义的训练与评估闭环

因此，下一步最合理的目标不是继续打磨当前 placeholder，而是按 `A -> B -> C -> D -> E` 的顺序，把这套最小工程框架逐步替换成真实系统。

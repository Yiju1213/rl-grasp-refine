# v5：论文目标系统、当前正式训练实现与剩余风险

本文件用于作为最终版前的系统状态文档。

它回答三个问题：

1. 论文 `docs/rl-regrasp/root.tex` 当前要求的总系统是什么。
2. 当前仓库里，哪些组件已经组成了可用于正式训练的系统。
3. 在已经可以正式训练的前提下，当前系统还存在哪些主要风险点。

本文件默认以当前代码为第一事实来源；若与 `v1.md` ~ `v4.md` 冲突，以当前代码为准。

## 1. 论文中的目标系统与模块要求

### 1.1 总系统定义

论文当前将任务定义为一个 **single-step in-contact grasp pose refinement** 系统。

其核心要求是：

- 一个 episode 只包含一次观测、一次 6-DoF corrective action、一次执行结果。
- 状态定义为 `s = [ z || t ]`。
- 动作为 `a = [Δp, Δθ]`，即 3 维平移增量和 3 维旋转增量。
- 策略 `π(a|s)` 通过 PPO actor-critic 优化。
- reward 由 `R_drop + R_contact + R_stability` 三项组成。

因此，论文要求的不是长时序 manipulation policy，而是一个围绕 **单步抓取位姿修正** 的感知-校准-奖励一体化系统。

### 1.2 感知状态要求

论文要求状态中的感知部分由两部分组成：

- `z`：由显式对齐的 visuo-tactile 3D observation 提取出的 latent feature。
- `t`：低维、可解释的 contact semantic。

其中 `t` 由两个量构成：

- `t_cover = |C| / |P|`
- `t_edge = (1 / |C|) * Σ min_{b in B} dist_tilde(p_i, b)`

论文语义是：

- `P` 是 tactile sensing points 的集合。
- `C = { p_i in P | d_i > τ }` 是有效接触点集合。
- `d_i` 是接触深度或形变量。
- `t_cover` 描述有效接触覆盖率。
- `t_edge` 描述有效接触到 sensor boundary 的归一化距离统计。

这意味着 contact semantic 在论文里不是附属 debug 指标，而是正式状态和正式 reward 的组成部分。

### 1.3 稳定性评估与在线校准要求

论文要求：

- 先由预训练稳定性预测器给出 raw stability logit：`ŝ = g_φ(z)`。
- 再通过两参数 logistic calibration 得到 calibrated probability：
  - `p = sigmoid(a * ŝ + b)`
- rollout 后使用 grasp execution outcome 对 `[a, b]` 做在线更新。
- 用 Hessian 的逆近似后验协方差：
  - `P = (H + lambda I)^(-1)`

这部分的工程含义是：

- 论文并不要求重训练 perception 主体。
- 论文要求在 online RL 过程中对稳定性估计做轻量但持续的分布对齐。
- reward 后续依赖的不是一个黑盒 uncertainty 分数，而是 `P` 及其标量化形式。

### 1.4 奖励要求

论文当前 reward 定义为：

- `R_drop`
  - 成功 `+1`
  - 失败 `-1`
- `R_contact`
  - `-lambda_c * max(0, tau_c - t_cover)`
  - `-lambda_e * max(0, tau_e - t_edge)`
- `R_stability`
  - `Delta p = p(g1) - p(g0)`
  - `R_stability = Delta p / (1 + kappa * tr(P))`

这表示：

- `R_contact` 是对 refined grasp 接触状态的 soft penalty，而不是 before/after 增量 reward。
- `R_stability` 依赖 calibrated probability 的相对提升，并且由 posterior uncertainty 衰减。

## 2. 当前系统中用于正式训练的组件

本节只描述当前已经接入正式训练主链、能支撑真实训练运行的组件。

### 2.1 场景与环境执行层

当前正式训练使用的是 dataset-backed、real-scene 风格的单步环境：

- `DatasetSampleProvider`
  - 从 tactile-extended dataset 中提供 sample 和对象/相机/抓取元数据。
- `PyBulletScene`
  - 重建 hand、object、camera、TACTO sensor。
  - 支持 live `before` / live `after` observation。
  - 支持 refine 后的 release trial。
- `GraspRefineEnv`
  - 把单步 refinement episode 组织成 `reset -> observe -> act -> refine -> observe -> trial -> done`。

因此，当前系统的 episode 语义已经和论文要求的 single-step refinement 形态一致。

### 2.2 显式对齐的 visuo-tactile perception 主链

当前 perception 主链已经是正式实现，而不是占位路径：

- visual 侧：
  - `depth + seg + proj + view -> world points`
- tactile 侧：
  - `depth + proj + gel pose -> gel/world points`
  - 保留 gel/contact mask
  - 构造 `sc_input` 和 `gs_input`
- backbone 侧：
  - 默认 `adapter_type: sga_gsn`
  - 使用 `SGAGSNPerceptionRuntime`
  - 调用 AdaPoinTr + SGSNet 组合得到 body feature 和 raw logit

因此，`z` 对应的显式 3D 对齐感知输入与预训练 backbone 路径已经进入正式训练闭环。

### 2.3 contact semantic、stability predictor 与 calibrator

当前 contact semantic 已经不再是 placeholder：

- `ContactSemanticsExtractor`
  - 从 `raw_obs.tactile_data["contact_map"]` 读取 tactile deformation magnitude
  - `t_cover` 由 active contact 占比得到
  - `t_edge` 由 tactile sensor 本地 2D 网格到最近边界的归一化距离均值得到

当前稳定性评估与在线校准也已经是论文方向实现：

- `ObservationBuilder`
  - 通过 `FeatureExtractor.encode(raw_obs)` 获取 `latent_feature`
  - 若 runtime 已提供 raw logit，则直接使用
  - 否则由 feature-only `StabilityPredictor` 从 `z` 预测 raw logit
- `OnlineLogitCalibrator`
  - 维护两参数 `a, b`
  - `predict(logit)` 输出 calibrated probability
  - `posterior_trace()` 提供 `tr(P)`
  - rollout 后用 `raw_logit_after + drop_success` 更新 `[a, b, P]`

这意味着论文中的 `ŝ -> p -> P` 这条稳定性信号链已经在正式系统中存在。

### 2.4 论文对齐后的 reward 实现

当前 reward 已经按论文结构收敛为三项：

- `R_drop`
- `R_contact`
- `R_stability`

其中：

- `R_drop` 直接由 release trial 成功/失败给出。
- `R_contact` 直接对 refined grasp 的 `t_cover / t_edge` 做 soft-hinge penalty。
- `R_stability` 使用 calibrated `p_before / p_after` 和 `posterior_trace` 计算。

当前环境里的 reward 计算时序是：

- rollout 内使用已广播到 worker 的 calibrator snapshot 计算 `R_stability`
- rollout 结束后主进程更新 calibrator
- 下一轮 rollout 再广播新的 calibrator state

这条链路已经可以支撑正式训练，并且与当前多进程训练架构兼容。

### 2.5 policy、PPO 与 observation contract

当前 RL 侧已经具备正式训练所需的组件：

- `PolicyObservationSpec`
  - 统一管理 policy flatten 顺序与 obs dim
  - 支持 `current / paper / no_pose / no_logit` preset
- `ActorCritic`
  - policy / value 网络共用 resolved observation spec
- `PPOAgent`
  - 在主进程完成 update
- `RolloutBuffer`
  - 保存单步 transition、log_prob、value、raw logits 和 infos
- `Trainer`
  - 负责 rollout collection、PPO update、calibrator update、logging

这使得系统已经具备从真实 observation 到真实 PPO 更新的完整训练主干。

### 2.6 正式多进程训练与运行环境组件

当前正式训练已经不是单环境脚本级能力，而是完整的多进程训练能力：

- `SubprocAsyncRolloutCollector`
  - `spawn` 启动
  - 每个 worker 固定单 env
  - worker 持有 local env、local calibrator snapshot、只读 actor 副本
  - episode 异步回传，主进程统一收集 batch
- `train.py`
  - 主进程持有 authoritative learner
  - 负责 actor-critic、optimizer、PPO、calibrator、checkpoint、logger
- `render_env.py`
  - 在 headless 场景下自动准备 `PYOPENGL_PLATFORM=egl`
  - 保证多进程 worker 初始化 TACTO offscreen renderer 时的基础环境一致

因此，当前系统已经具备：

- real scene
- SGA-GSN
- online calibrator
- PPO
- 多进程单 env 异步采样

这一整套正式训练能力。

## 3. 当前系统仍然存在的主要风险点

### 3.1 policy 默认状态仍不是论文定义的 `s = [z || t]`

虽然系统已经支持 `paper` preset，但默认训练配置仍是 `current`：

- `latent_feature`
- `contact_semantic`
- `grasp_pose.position`
- `grasp_pose.rotation`
- `raw_stability_logit`

所以从论文严格对齐角度看，当前 policy 输入仍然比论文定义更大。

这不是系统不能训练的问题，而是最终论文收敛前仍需要决定是否切到 `paper` preset 作为正式默认值。

### 3.2 calibrator 与 reward 的时序仍是 snapshot 语义

当前 reward 使用的是 rollout 开始时广播给 worker 的 calibrator snapshot。

这有两个工程优点：

- 保持 env/worker 内 reward 可直接计算
- 不需要在 batch 后回填 rewards

但它也带来一个残余差异：

- `R_stability` 使用的 `tr(P)` 不是“同一 rollout 收集完后再更新得到的 posterior”
- 而是上一轮同步下来的 posterior snapshot

因此，当前实现是因果一致、工程稳定的 online 版本，但不是最严格的“same-batch posterior reward”语义。

### 3.3 `t_edge` 是合理工程化定义，但仍不是唯一可能定义

当前 `t_edge` 已经不再是 placeholder，但它的具体工程定义是：

- tactile sensor 本地 2D 网格
- 到最近边界的归一化距离均值

这与论文语义一致，也足够可解释，但论文并没有唯一写死 `dist_tilde` 的具体工程实现。

因此，当前 `t_edge` 已经可以视为正式实现，但若论文最终想采用更强物理解释的 boundary metric，仍有再次收敛的可能。

### 3.4 多 worker 下 SGA-GSN runtime 仍是逐进程复制

当前多进程训练时：

- 每个 worker 都会各自加载 local perception stack
- `SGAGSNPerceptionRuntime` 只做进程内共享，不做跨进程共享

这意味着风险仍然存在于：

- GPU 显存占用近似随 worker 数增长
- TACTO + perception + renderer 在同一张 GPU 上会产生 contention
- worker 数进一步增大时，吞吐不会线性提升

因此，当前多进程训练能力已经“可正式使用”，但“最佳 worker 数”仍然属于部署与资源调优问题。

### 3.5 运行环境仍然依赖 headless OpenGL / EGL 条件

当前代码已经能在 `use_gui=False` 时自动准备 `PYOPENGL_PLATFORM=egl`，但这仍然意味着：

- 训练机器需要具备可用的 EGL / offscreen 渲染条件
- TACTO / pyrender / OpenGL backend 的系统依赖仍然是正式部署条件的一部分

所以这一风险已经从“代码未处理”变成“系统部署条件必须满足”。

### 3.6 reward 更换后，训练超参数可能仍需重新整定

当前系统已经从工程近似 reward 切换到了论文对齐版 reward。

这意味着即便训练主链已经能跑：

- PPO 学习率
- batch size
- worker 数
- `kappa`
- `lambda_c / lambda_e`
- `tau_c / tau_e`

这些值仍可能需要重新调优，才能得到更稳定或更优的训练结果。

这属于系统已经闭环、但最终性能尚未完全收敛的风险。

### 3.7 论文文本自身仍然不是完全定稿状态

当前 `root.tex` 仍带有明显草稿属性：

- abstract 尚未写完
- introduction / experiments / conclusion 仍在收敛
- 个别公式与 appendix 之间仍存在表述简化

因此，当前系统已经足以作为论文最终系统的工程基础，但论文文本本身仍有继续定稿的空间。

## 4. 当前结论

如果只回答一句话，那么当前系统可以概括为：

> 当前仓库已经具备以真实 scene、真实 SGA-GSN 感知、在线 calibrator、论文对齐版 reward、PPO 和多进程异步采样为核心的正式训练系统；在此基础上，剩余工作主要集中在论文最终状态定义收敛、资源/部署调优，以及若干工程辅助模块与系统精简性重构。

因此，`v5.md` 可以视为最终版前的主状态文档，而后续版本更适合只记录：

- 辅助模块补充
- 默认配置收敛
- 系统瘦身与重构
- 文档与实现的最终同步
